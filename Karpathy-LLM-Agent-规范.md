# Karpathy 编码指南（精简中文版）
减少 LLM 常见编码错误的提示规范。可与项目自有规范合并使用。

**权衡：** 偏向谨慎而非速度。琐碎任务自行判断是否省略。

---

## 1. 先想再写

**不要假设。不要掩饰困惑。暴露权衡。**

动手前：
- 明确说出你的假设。不确定就问。
- 存在多种理解时摆出来，别自己悄悄选一个。
- 有更简单的方案就说出来，该反驳时反驳。
- 不清楚就停下来，说出哪里困惑，然后问。

## 2. 简单至上

**解决问题的尽量少代码。不要投机代码。**

- 不要做需求之外的功能。
- 只用一次的函数不要抽象。
- 没人要求的灵活性和可配置性不要加。
- 处理不可能发生的错误是浪费。
- 写了 200 行但 50 行能搞定的，重写。

问自己："资深工程师会觉得这代码过度复杂吗？"是就简化。

## 3. 精确改动

**只动你该动的。只清理你造成的垃圾。**

改老代码时：
- 不要顺手"优化"相邻代码、注释或格式。
- 没有坏的东西就不要重构。
- 沿用现有风格，哪怕你不喜欢。
- 看到无关的死代码可以提一嘴，但别删。

你的改动产生了孤儿：
- 只删你的改动导致不再使用的 import/变量/函数。
- 别动本来就存在的死代码，除非有人要求。

检验标准：每个改动的行应该能直接追溯到用户需求。

## 4. 目标驱动

**定义成功标准。循环直到验证。**

把任务转化成可验证的目标：
- "加校验" → "先写无效输入的测试，再让测试通过"
- "修 bug" → "先写复现测试，再让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步任务写出简要计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

好的成功标准让你能独立迭代。差的标准（"弄好它"）需要不断找人确认。

---

**这些指南生效的表现：** diff 里不必要的改动变少，因过度复杂而重写的次数减少，疑问在写代码前提出而非犯错之后。
