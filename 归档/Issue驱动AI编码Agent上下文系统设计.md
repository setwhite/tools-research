# Issue/PR 驱动的 AI 编码 Agent 上下文系统设计

## 1. 上下文从哪来？

AI 编码 Agent 每次被唤醒，必须回答一个问题：**现在要做什么？**

最可靠的信息不在聊天记录里，在 GitHub Issue 和 PR 里。标题描述了意图，正文定义了当前行为与目标行为的差距，评论线程记录了人类协作的决策痕迹。这些东西持久、可寻址、变更可审计——恰好满足"单一事实源"所需的全部属性。

**所以设计原则很直白：Issue 是规范文档，不是聊天记录。Agent 每次执行前重新读 Issue，从最新状态重建上下文，不依赖"上次聊到哪了"。**

---

## 2. 重建流水线

一条完整的执行周期分四步：

**Step 1：理解（Understanding）**

Agent 读 Issue 正文和最新评论 → 生成 Specification。Specification 包含两部分：
- Current Behavior：通过对代码库的实际阅读得出的当前行为
- Proposed Behavior：对 Issue 目标的精炼表述，附带可验证的成功标准

Specification 是第一个人类干预点——在 Agent 写代码之前，先确认它理解对了。

**Step 2：规划（Planning）**

Agent 定位相关文件，生成精确到文件级别的变更清单——哪些文件要改、每个文件做什么。Plan 是第二个人类干预点。

**Step 3：执行（Execution）**

按 Plan 顺序实施修改。

**Step 4：交付（Delivery）**

创建 PR，描述中自动引用 Specification 和 Plan，关联原 Issue。

整个流程的核心设计是**两个 Steering Point**（Spec + Plan），确保人类可以在最关键的节点纠偏，而不是等 Agent 闷头干完再推倒重来。

---

## 3. 为什么不并行？

直觉反应是"拆成子任务，分给多个子 Agent 并行"。这在代码任务中是一个已知的反模式。

**原因一：子 Agent 缺少完整上下文。** 主 Agent 在执行中积累了上游决策——为什么选了方案 A、接口为什么这样设计——只给子 Agent 一句"去写 XX 功能"，它会在完全不同的假设下工作。

**原因二：并行决策会互相冲突。** 两个子 Agent 各自做 UI 的不同部分，各自选不同的组件库和交互模式——单独看都没错，合并起来不可用。

Cognition（Devin 团队）的结论：**单线程线性 Agent。** Claude Code 的子 Agent 也永远只回答问题，不写代码。这不是缺乏创意——在上下文共享问题被根本解决之前，多 Agent 并行引入的决策冲突比它解决的问题更多。

---

## 4. 成本：值得吗？

反对 Issue 驱动的最常见论据是 Token 开销——每次重建上下文确实要多花几千 Token。

但对比一下：一个跑了 200 轮 tool call 的旧上下文，模型需要花大量 Token 去理解"上次发生了什么"，而这种理解还经常出错。出错后的纠偏轮次、被打回的 PR、重新讨论和重新执行——这些成本远超那几千 Token。

**多花的 Token 是为上下文的正确性买保险。** 在工程领域，"不容易出错"远比"快"宝贵。

---

## 5. 现实中需要注意的

**Issue 质量决定成功率。** 只有一行标题"修复登录 bug"的 Issue，Agent 和人类一样抓瞎。推广 Issue 模板——当前行为、期望行为、复现步骤、相关文件——不仅是服务 Agent，更是基本的工程素养。

**评论是双刃剑。** 被讨论 50 条的 Issue，前 30 条可能已被后 20 条推翻。全量读取等于给模型喂矛盾信号。按时间倒序读取并设上限，或先做评论摘要，比无脑全读更安全。

**代码库可发现性是前提。** 文件命名混乱、无测试、无模块边界——同样的信息障碍，人类接手也抓瞎。这不是上下文设计的问题，是代码库本身的问题。

---

## 6. 结语

Issue 驱动的上下文设计，本质是把**软件工程已有的协作基础设施**对 Agent 开放，而不是为 Agent 发明一套新信息架构。

选择 Issue 作为事实源意味着：Agent 和人类共享同一套信息管道；每次执行从当前真实状态开始，而非上次遗留状态；上下文的正确性由外部可审计的事实源保证。

这不是最快的方案。是目前已知最不容易出错的方案。
